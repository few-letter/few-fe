설계 제안 시 책임·메시지·협력 기준을 적용합니다.

**참고**: `.claude/OOP_THINKING_GUIDE.md`의 원칙을 기반으로 설계합니다.

## 설계 원칙

### 핵심 질문
```
❌ "어떤 클래스가 필요한가?"
✅ "어떤 객체들이 어떤 메시지를 주고받으며 협력하는가?"
```

### 책임 주도 설계 (RDD)
1. 시스템 기능 파악
2. 기능을 더 작은 책임으로 분해
3. 책임을 수행하기에 적합한 객체에 할당
4. 객체가 외부에 제공할 책임을 먼저 결정
5. 그 책임 수행에 필요한 데이터를 나중에 결정

### 책임의 두 유형
| 유형 | 설명 |
|------|------|
| Knowing | 자신의 상태, 관련 객체, 계산 가능한 정보를 아는 것 |
| Doing | 객체 생성, 연산, 다른 객체에 작업 요청 |

### 구조 vs 기능
- "기능"보다 "구조"가 더 안정적
- 변경을 예측하지 말고, 변경을 수용할 수 있는 설계
- 좋은 설계 = 기능 변경 시 코드 변경 범위가 좁음

## 출력 형식

1. **도메인 모델**: 핵심 개념과 관계 (텍스트 다이어그램)
2. **책임 할당표**: 각 객체의 Knowing/Doing 책임
3. **협력 흐름**: 메시지를 통한 객체 간 상호작용
4. **코드 스케치**: 인터페이스/클래스 구조 (필요시)
